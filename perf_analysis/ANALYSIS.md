# Research pipeline: куда уходит время (анализ 2026-06-12)

**Вопрос:** какие основные потребители времени в ресёрче (serp / scrape / LLM /
подбор прокси), где бутылочные горлышки, что делать. Плюс — контекст LLM на
каждом шаге для планирования batching / уменьшения ctx / повышения np.

**Данные:**
1. `mine_worker_log.py` — wall-time-разложение taskiq-worker-лога прогона optikov
   2026-06-11 (11:18–23:59, 77 завершённых + 23 таймаута, concurrency=3).
   Каждый worker-процесс ведёт одну задачу за раз (проверено: интерливинг 0.1%),
   поэтому интервалы между строками лога атрибутируются корзинам.
2. `mine_llama_log.py` — per-request prefill/decode из llama-server-лога той же
   сессии (1347 запросов).
3. `aggregate_card_perf.py` — точная per-call телеметрия из новых карточек
   (инструментирование агента, коммиты `436924a`+`22123c2`; банкуется прогоном
   23 недостающих оргов optikov).

## 2. Внутренности llama (сессия прогона)

- busy 14.1 ч (клиентских 16.0 ч — остальное сеть/JSON), **94% busy = decode**,
  prefill всего 6%.
- **Decode ~8.1 tok/s per-stream** при np=3 (одиночный поток на этой машине
  ~42 tok/s; в менее контендed сессиях 31–38). Контеншн трёх потоков.
- Prefill 843 tok/s; нового префилла в среднем 1939 ток/запрос → **KV-reuse ~79%**
  (in-slot кэш работает; KV-paging строить незачем — см. design-notes).
- Контекст per-request: med 9.0k, p90 16.1k, max 23.3k — при 36k/слот
  **запас 35%+ даже на p90**; compaction = 0.
- Completion: med 173, p90 625, **p99 1530 токенов** (reasoning-хвост).

## 3. Механика главного горлышка (38% сгоревшего времени)

Цепочка: reasoning-модель периодически выдаёт длинные completion (p95+ ≥ 900
ток.) → при contended decode ~8 tok/s это >110–190s → часть упирается в
`RESEARCH_LLM_TIMEOUT_S = 180` → клиент рвёт соединение, **llama отменяет запрос
(весь decode выброшен)** → клиент ретраит с нуля. В llama-логе 0 запросов >180s
(max 179.9) — это не совпадение, это обрезка. Ретраи **хронические** (10–29% во
все часы, ~16% вызовов), не деградация к концу.

Каждый таймаут = до 180s слота себе + замедление двух соседей (декодят
одновременно) → вторичные таймауты. Отсюда и 23 карточки-таймаута (>60 мин/орг).

## 4. SERP: 60% пустых ответов

SearXNG (локальный) с ретраями 4×~8s+0.5s: пустой organic в 60% попыток прогона
(и 43% в сегодняшнем). Фейловый web_serp стоит ~33s. Это здоровье upstream-движков
(баны/рейт-лимиты), не наш код. В свежем прогоне при быстрой llama serp временно
становится потребителем №1 (~45% на первых карточках).

## 5. Рейтинг рычагов

1. **Поднять LLM-таймаут до 420–600s** (`RESEARCH_LLM_TIMEOUT_S`) — одна строка.
   Убирает выбрасывание работы: возвращает бóльшую часть 38%. Хвостовой запрос
   лучше дождаться за 200–300s, чем дважды пересчитывать.
2. **Ограничить completion** (`max_tokens` ~800–1000 на main/critic) — режет
   p99=1530 хвост, который и пробивает таймаут. Дёшево, но проверить качество.
3. **Decode-скорость**: np=3→2 (быстрее per-stream, но -1 слот — нетто мерить);
   спекулятивный декодинг с draft-моделью (большой потенциал именно на длинных
   reasoning-цепочках); `-c 108000` можно ужать до ~72k (24k/слот ≥ max 23.3k) —
   освобождает VRAM, но декод это ускорит слабо.
4. **SearXNG**: разобраться с движками (какие отдают пусто, рейт-лимиты),
   снизить попытки 4→2 (терять нечего — 60% пустых это уже фейлы), кэш SERP.
   Потенциал ~7% + ускорение шагов агента.
5. **Прокси**: dead-skip в браузер-путь (реюз `_DEAD_PROXIES` из httpx-пути),
   короче goto-timeout первой попытки. Потенциал ~2–3% — приятно, не критично.
6. **Батчинг/KV-paging — не строить** (подтверждено и этим замером: prefill 6%
   busy, reuse 79%; вся боль в decode и таймаутах).

## 7. Два режима и развязка причины

llama-сессии при ИДЕНТИЧНОМ конфиге (np=3, c=108000, q8 KV) и той же
конкурентности:

| | 11.06 (деградировавший) | 12.06 (свежий) |
|---|---|---|
| decode tok/s (per-stream) | **8.1** | **37.8** (×4.7) |
| запрос med / p90 | 24.3s / 88.9s | 7.2s / 25.5s |
| таймауты LLM (>180s) | ~16% вызовов | 0 |
| итог per-org | med ~19 мин, 23/100 таймаутов | 3.4–10.8 мин, 0/23 |

ctx и длины completion в обеих сессиях сопоставимы → **слоулайн 11.06 — не
np-контеншн и не контекст, а деградация среды** (12ч аптайм под полной
нагрузкой: давление RAM от 8 воркеров×Playwright, возможная WDDM-демоция VRAM,
троттлинг — слои загрузки в error-лог не пишутся, ретроспективно не различить).
Каскад был вторичен: медленный decode → хвост >180s → ретраи выбрасывают
работу → ещё медленнее.

**Известный прецедент (контекст от оператора):** такая же деградация ранее
наблюдалась в LM Studio (concurrency=2, ctx 100k q8, llama CUDA 12.0 при
системной CUDA 13.0): в Task Manager **VRAM не менялась, загрузка CUDA резко
падала, а метрика «3D» подскакивала** — и decode проседал ровно на наблюдаемую
здесь величину. При этом 13-часовой прогон на 517 ресёрчей (llama, np=3,
ctx=205k, kv=q4 — заметно более медленный конфиг) деградации не показал.
Гипотеза «движок/конфиг-зависимая WDDM-демоция при q8/высокой скорости» —
**причину НЕ чинить вслепую**: ловим повторения на текущем конфиге и
диагностируем (см. watchdog ниже — он снимает nvidia-smi снапшот в момент
срабатывания, `llama_degradation_snapshots.log`).

**РАЗВЯЗКА — поймано вживую 2026-06-12 ~16:20 (старт кировского батча):**
на этот раз не «медленно», а полный клин: запросы НЕ завершаются (клиент
отменяет по таймауту — в llama-логе серия `stop: cancel task`), `/health`
отвечает 200, `/metrics` висит, монитор замер (cum_n не растёт). GPU-улики
(сняты до рестарта, в `llama_degradation_snapshots.log`):
- nvidia-smi: util.gpu **100%**, util.memory (шина VRAM) **0%**, 87W, P2 —
  «занят, но не работает»;
- per-process engines: llama cuda-engine **98.8%**; 3D-engine жгут
  **dwm.exe 38.5% + csrss.exe 32.2% + Claude.exe(gpu-process) 29.4%** —
  обычный десктоп-композитинг, не Playwright;
- **llama GPU memory: local 7226MB + shared 296MB** ← дымящийся пистолет:
  WDDM вытеснил часть CUDA-аллокаций в системную RAM. Каждый шаг декода
  читает эти страницы через PCIe (~12GB/s vs 448GB/s VRAM) — отсюда
  cuda 99% busy при нулевой шине VRAM и низком потреблении. Полное
  совпадение с LM Studio-сигнатурой («VRAM flat, CUDA падает, 3D растёт»).
Механика: llama держит 7.2GB на 8GB карте, которая одновременно рисует
десктоп; всплеск VRAM-аппетита композитора/приложений → WDDM выселяет
страницы CUDA → PCIe-молотилка. Рестарт сервера перевыделяет память и
возвращает скорость (та же «лечилка», что и 11.06/12.06).

**Вторая находка (та же сессия): pm2 restart на Windows плодит сирот.**
После рестартов обнаружились **ТРИ живых llama-server.exe одновременно**
(16:38, 16:43, 16:43) — pm2 не убил заклинивший процесс и запустил новый
рядом; старые держали VRAM → каждый свежий инстанс рождался сразу в
демоцию (prefill 23 tok/s vs 843 здоровых — замер из лога). Хронический
цикл «рестарт → 3 мин → клин» был самоподдерживающимся из-за сирот.
Лечение: `taskkill /IM llama-server.exe /F` ПЕРЕД `pm2 restart` —
встроено в watchdog v2.

Применённый фикс (решение оператора 2026-06-12 ~17:00): **`-c` 108000 →
81000** (27k/слот, q8_0; p99 вызовов < 24k) + **`RESEARCH_COMPACT_TRIGGER_
TOKENS` 50000 → 23000** (.env; раньше порог был выше слота и не работал) —
освобождает ~540MB VRAM как запас против демоции. ecosystem.config.js
заодно приведён к фактическому конфигу (в файле лежал устаревший
q4_0/195k, реальные args жили только в кэше pm2).
Оставшиеся кандидаты (не применены): NVIDIA Control Panel → «CUDA - System
memory fallback policy» → «Prefer No System Memory Fallback» (вместо тихой
демоции — громкий OOM); глубже ужать -c под draft-модель speculative.

**Следствия для рычагов** (уточнение §5):
1. Таймаут 180→420–600s — всё ещё №1: он превращает «деградация → лавина
   выброшенной работы» в «деградация → просто медленнее». Дёшево, без минусов
   для здорового режима (там p99=171s < 180). (Пока НЕ применяем — см. ниже.)
2. **Детекция деградации — РЕАЛИЗОВАНО (2026-06-12)**:
   `yandex_enrichment_experiment/llama_watchdog.py` под pm2 (`llama-watchdog`) —
   rolling decode tps по eval-строкам llama-лога (окно 10 мин, ≥8 запросов);
   при **<11 tps** снимает GPU-снапшот (nvidia-smi state + throttle reasons →
   `llama_degradation_snapshots.log`) и делает `pm2 restart llama-server`
   (cooldown 15 мин). Конфиг llama при этом не меняется — цель поймать и
   задокументировать деградацию, а не маскировать её.
   **v2 (после клина 16:20)**: tps-порог слеп, когда запросы вообще не
   завершаются (нет eval-строк → окно пустое). Добавлен wedge-probe:
   `/health`=200 при висящем `/metrics` 2 проверки подряд → снапшот+рестарт.
   Снапшот расширен per-process GPU-engine разбивкой и local/shared GPU
   памятью (улики WDDM-демоции).
   Монитор (`monitor_llama_ctx.py`) дополнен `dec_tps` и режимом
   `MONITOR_FROM_END=1` (без затравки историей). Графики всех метрик во
   времени: `perf_analysis/plot_monitor_metrics.py`; распределения контекста
   по типам вызовов: `perf_analysis/plot_ctx_by_kind.py`.
3. SearXNG — теперь главный рычаг здорового режима (~40%): разобрать движки
   с пустыми ответами, попытки 4→2, кэш SERP.
4. Прокси dead-skip в браузер-путь — 6.3% здорового режима, стоит сделать.
5. Батчинг/KV-paging/ужатие ctx — подтверждённо НЕ рычаги (prefill 6–15%
   busy, reuse 79%, мелкие вызовы ~7% времени).
