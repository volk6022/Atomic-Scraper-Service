# Экономия трафика при парсинге Яндекс.Карт — итоги экспериментов

Цель: уменьшить стоимость (резидентские прокси ~$3/ГБ). Исходный прогон стоил
бы ~$104 (15 025 запросов × 2.37 МБ), баланс 1.6 ГБ кончился на ~744 запросах.

## Что измерено

### Эксп. 1 — оптимальный шаг сетки (`01_grid_sim.py`, без трафика)
На плотно отснятом центре (31 ячейка, радиус ~570 м) приняли найденное за «правду».
- **Потолок выдачи ~24–25 орг/запрос почти универсален.** Сверх него вылезают только
  плотные категории («кафе» 49, «салон» 33, «магазин» 32). → **зум-аут полноту НЕ
  повышает** (тот же ~24, размазанный по большей площади).
- recall vs шаг: 180 м=100%, **~280 м=95%**, 360 м=91%, 540 м=69%.
- → шаг 180 м избыточен (overlap-фактор был 15.5×). Разумный диапазон **250–300 м**.

### Эксп. 2 — трафик на 1 запрос в браузере (`02_traffic_probe.py`, живой)
| политика | трафик | орг | капча |
|---|---|---|---|
| baseline (как прод) | 2369 КБ | 24 | нет |
| режем картинки | 2544 КБ | 24 | нет |
| **+ css + хосты тайлов/ястатик** | **511 КБ** | 24 | нет |
| **+ без скролла** | **483 КБ** | 24 | нет |
| **только документ** | **525 КБ** | 24 | нет |

Данные целиком в inline-SSR HTML. JS-бандл карты (`maps.yastatic.net` 1.15 МБ),
тайлы и фото — чистый оверхед рендеринга. Блокировка даёт **4.6×**.

### Эксп. 3–4 — БЕЗ браузера, httpx GET (`03_*`, `04_*`, живой)
**16/16 запросов: 200, капчи 0, по 25 орг, в среднем 343 КБ/запрос** (gzip по проводу),
через 16 разных IP / 4 точки / 4 категории.
- кафе ~500 КБ, стоматология ~280 КБ, аптека ~190 КБ.
- Это и дешевле браузера, и параллелится, и убирает Chromium/CPU.
- Минус: только SSR-страница (25 орг), пагинации нет — компенсируется частотой сетки.

## Проекция стоимости на круг 2500 м (25 категорий)

| шаг | ячеек | запросов | браузер (2.37 МБ) | **httpx (0.34 МБ)** | recall |
|----:|------:|---------:|------:|------:|---:|
| 180 м | 601 | 15 025 | $104 | $15 | 100% |
| 250 м | 317 | 7 925 | $55 | **$8.0** | ~96% |
| 300 м | 221 | 5 525 | $38 | **$5.6** | ~93% |
| 360 м | 145 | 3 625 | $25 | $3.6 | 91% |

## Рекомендация

1. **Отказаться от браузера в discovery** — перейти на httpx GET + парс SSR-скрипта.
   Один общий клиент, `Accept-Encoding: gzip`, ротация тех же HTTP-прокси. ~7× дешевле
   на запрос, капча не замечена (но мониторить долю не-200/капчи и держать фолбэк
   на браузерный путь). Отзывы (`reviews`) по-прежнему требуют браузера (replay сессии).
2. **Шаг сетки 250–300 м** вместо 180 м (~4–6× меньше запросов при recall 93–96%).
3. **Без пагинации** (только SSR, 25 орг) — для редких категорий потерь нет, для
   плотных компенсируется шагом сетки.
4. (Опц.) **Категориально-адаптивный шаг**: плотным («кафе», «магазин», «салон»,
   «парикмахерская») — 200 м; остальным ~22 — 400 м. Ещё экономнее.

**Итог:** httpx + сетка 300 м ≈ **$5–6** и ~93% полноты против исходных ~$104.
То есть полный круг можно отснять примерно за **1.8 ГБ** вместо ~36 ГБ.

# Отзывы (эксп. 5–8)

Старый endpoint `reviews` был нестабилен: он наблюдал XHR `fetchReviews` и **реиграл**
URL через `page.request.get()`. Параметры `s`/`reqId` одноразовые/протухающие → сбои.

### Что нашли
- **Отзывы лежат в SSR-HTML** страницы `/reviews/`, путь
  `stack[0].results.items[0].reviewResults.reviews` — **50 отзывов в первой пачке**,
  ~190 КБ, без капчи. Полная схема (автор, текст, рейтинг, ответ бизнеса, фото, реакции).
- Рядом `reviewResults.params`: `count, totalPages, page, reviewsRemained` — и
  **`csrfToken` прямо в SSR**.
- **Пагинация чистым httpx через `?page=N`** (`05–07`): страницы дают по **50
  полностью непересекающихся** отзывов, ~160–190 КБ/стр (**~3.4 КБ/отзыв**), **без
  браузера, без csrf, без подписи `s`**. (URL-ключ именно `?page=`; `reviews[page]`
  не работает.)
- **Лимит глубины SSR: ~12 страниц** (страницы 1–12 отдают по 50, с ~15-й — пусто).
  Т.е. httpx достаёт **~600 отзывов/орг** — хватает подавляющему большинству;
  для гигантов (рестораны с тысячами) хвост — браузерным фолбэком.
- Прямой вызов `/maps/api/business/fetchReviews` через httpx даёт **400** — требует
  клиентскую подпись `s` (считается JS). Реверсить не нужно — `?page=N` это обходит.
- **Браузерный путь как фолбэк** (`08`): не replay, а **перехват ответов** `fetchReviews`
  при скролле (браузер сам считает `s`) + блокировка тайлов/картинок → 200 отзывов,
  без капчи, 8.3 КБ/отзыв (фикс. JS-бандл размывается на глубине).

### Рекомендация по отзывам
1. **Переписать discovery отзывов на httpx + `?page=N`** — стабильно, дёшево
   (~3.7 КБ/отзыв), параллелится, без браузера и без хрупкого replay.
2. Глубина: цикл `?page=1..12` до пустой страницы (~600 отзывов потолок httpx).
   Первая страница (50) почти всегда достаточна для аналитики.
3. Для орг с >600 отзывов и при капче — браузерный фолбэк с **перехватом** ответов
   `fetchReviews` при скролле (НЕ replay) + блокировка тайлов/картинок.

# Отзывы «последние 6 мес / макс 300» + карточка организации (эксп. 9–11)

### Отзывы за период (`10`)
- `updatedTime` — ISO (`2026-06-01T21:18:16.594Z`), парсится тривиально.
- `?page=N&ranking=by_time` — свежие первыми. Стоп: набрали 300 ИЛИ страница
  целиком старше cutoff. Порядок между страницами не строго монотонный — поэтому
  останавливаться надо на **полностью** просроченной странице, а не на первой
  «пограничной».
- Пример (Счастье, 9586 отзывов всего): **за 6 мес всего 84 отзыва**, 2 страницы,
  378 КБ. Фильтр по дате почти всегда держит объём << 300.

### Карточка организации `/maps/org/{seoname}/{oid}/` (`09`, `11`)
Наблюдение «при клике даёт больше» подтвердилось частично. Один GET ~**100 КБ**.
- Поиск (SSR) уже отдаёт **~67 полей**, карточка — **70**. Богатые атрибуты
  (`features`: средний счёт, цены, оплата, навынос, онлайн-запись, доступность)
  **уже есть в нашей выдаче** — 1135/1143 орг с features.
- **Только в карточке / полнее**:
  - `socialLinks` — VK / WhatsApp и т.п. (в нашей search-выдаче пусто!) ✅ ценно
  - `neurosummaryData` — AI-саммари из репрезентативного отзыва ✅
  - `relatedPlaces` — ~8–10 похожих/конкурентов ✅
  - `description`, полнее `address`, `parentRubrics`
  - встроено ~3 превью-отзыва (полные 50 — на под-странице `/reviews/`)
- Важно: «лайт»-объекты пользователь видел при скролле карты — это XHR-пагинация
  `/maps/api/search`. **SSR-первая страница отдаёт «толстые» объекты** (мы их и берём).

### Рекомендация
- **Отзывы:** httpx `?page=1..N&ranking=by_time`, стоп по 6 мес / 300. ~4.5 КБ/отзыв.
- **Обогащение:** один GET карточки (~100 КБ) добирает `socialLinks`, AI-саммари,
  `relatedPlaces`, `description`. Имеет смысл дёргать только для нужных орг, т.к.
  атрибуты уже собраны поиском.

Трафик, потраченный на все эксперименты: ~30 МБ.
