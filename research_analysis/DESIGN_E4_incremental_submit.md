# E4 — дизайн: инкрементальный submit-and-continue (черновик на обсуждение)

## Задача (от пользователя)
Сделать НОРМАЛЬНЫМ поведение, когда агент сабмитит карточку сразу по мере получения
данных и продолжает работу (дозаполняет). Но при этом — НЕ уходить в дебри и НЕ
упираться в лимит 25 ходов.

## Текущее поведение
`submit_result` — терминальный. Один большой сабмит в конце → критик оценивает →
при reject агент продолжает (до `max_rejects`, потом force-accept). Т.е. цикл
submit→continue уже есть, но только через ОТКЛОНЕНИЕ критика, и submit мыслится
как «финал». Плюс уже добавлен форс-сабмит на выходе (W1-фикс).

## Предлагаемый дизайн: «running card» + не-терминальный submit с флагом final

1. **Стартовая карточка предзаполнена** данными Я.Карт (E1/E2: соцсети, телефоны,
   часы, рейтинг, отзывы). Агент это видит как ДАНО. Его задача — закрывать ПРОБЕЛЫ.

2. **`submit_result(result, sources, final=false)`** становится не-терминальным:
   - сервер МЕРЖИТ `result` в running-card (не затирая непустое прежним пустым);
   - запускает критика;
   - ВОЗВРАЩАЕТ агенту: `critic_feedback` + **список оставшихся пробелов**
     (пустые поля схемы) + **сколько ходов осталось**;
   - инструкция: «есть пробелы и ходы — доисследуй и сабмить снова; всё закрыто —
     `submit_result(..., final=true)`».
   - При `final=true` → терминирование с merged-card.

3. **Анти-дебри / анти-лимит — три предохранителя:**
   - **Пробелы как явная цель.** Список пустых полей сокращается → понятный критерий
     остановки. Пусто пробелов → авто-финал.
   - **Счётчик «без нового».** Если K ходов подряд (напр. K=3) не добавили НИЧЕГО
     нового в running-card → авто-финал (агент топчется).
   - **Бюджет ходов виден + хард-стоп.** Каждый ход сообщаем «ход 8/12». На `cap-1`
     — принудительный финал из running-card (форс-сабмит уже есть).

## Почему это решает «дебри/лимит»
Сейчас агент уходит в дебри, потому что у него НЕТ явного критерия «готово» кроме
оценки критика. Running-card + список пробелов дают детерминированный критерий:
работа = закрыть пробелы; пробелов нет ИЛИ прогресса нет ИЛИ ходы кончились → стоп.
Предзаполнение из Я.Карт сразу закрывает соц/телефоны/часы → меньше пробелов →
короче цикл.

## Минимальная реализация (если ОК)
- `agent.py`: ввести `running_card` (dict), мерж-функция; в ветке submit — мержить,
  считать gaps (пустые ключи output_schema), вернуть gaps+remaining_turns вместо
  терминирования (кроме `final`/cap/no-progress).
- В `build_tools` добавить булев `final` в submit-тул.
- Промпт submit-тула: описать checkpoint-семантику.
- Метрики: turns, gaps_closed_per_turn — для проверки, что не топчется.

## Риски / открытые вопросы
- **Стоимость:** несколько критик-вызовов на ресёрч (по одному на checkpoint) →
  больше токенов. Митигация: запускать критика только на `final` или каждые N
  checkpoint-ов.
- **Мерж-семантика:** как мержить списки (sources/social) — конкатенация+дедуп;
  скаляры — не затирать непустое пустым. Надо зафиксировать правила.
- **Стоит ли вообще?** Если предзаполнение E1/E2 уже резко сокращает пробелы, может
  хватить текущего одно-сабмитного цикла + force-submit. A/B (E3) покажет, насколько
  prefill уже улучшает полноту — и нужен ли инкремент сверх него.

## Рекомендация по порядку
Сначала дождаться итогов A/B (E3): если prefill сам по себе закрывает большинство
полей за короткий цикл — E4 можно упростить (оставить одно-сабмитную модель с gap-
подсказкой и no-progress-стопом, без полноценного checkpoint-протокола).
