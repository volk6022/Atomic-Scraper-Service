# Эксперименты по оптимизации скрапинга — ВЕРИФИЦИРОВАННЫЕ результаты

**Дата:** 2026-06-09. Метод: реальные URL из прогона 132 орг прогнаны 3 способами
(httpx+SSR / браузер с resource-block / браузер как сейчас) через тот же резидентский
пул `proxies.txt`. Байты считаны по проводу (`request.sizes().responseBodySize` для
браузера, `num_bytes_downloaded` для httpx). Код: `exp_compare.py`, `probe_vk_2gis.py`,
`estimate_savings.py`. Сырые данные: `exp_results.json`.

## ⚠️ ГЛАВНАЯ ЛОВУШКА: Brotli (без него половина доменов = молчаливый мусор)
Эксперимент вскрыл то, чего в плане нет вообще. В venv **не установлены `brotli`/
`brotlicffi`**. httpx по умолчанию шлёт `Accept-Encoding: …, br`; домены за CDN с
brotli (zoon, rubrikator, spravker, **2gis**) отвечают `Content-Encoding: br`, и httpx
**не может это распаковать** → `resp.text` = бинарный мусор. Но `count_words()` дробит
мусор на «слова» и даёт **высокий word_count** — метрика выглядит зелёной, а контента нет.
Доказано (`probe_brotli.py`): тот же URL с `Accept-Encoding: gzip, deflate` → `CE=gzip`,
чистый кириллический текст (ratio 0.39 → 0.98).

**Вывод для внедрения:** httpx-fetch ОБЯЗАН либо (а) **не слать `br`** (просто и надёжно),
либо (б) поставить пакет `brotli`. Без этого t.me/hh/prodoctorov/rusprofile/orgzz/2gis…
часть отдаст мусор. Это же касается существующего `yandex_maps._HTTP_HEADERS` (там тоже
`br` — yandex молча падает в браузерный fallback на br-ответах = лишний трафик).

**Важная оговорка по 2gis:** после фикса 2gis декодируется в ЧИТАЕМЫЙ текст, но это
**интерстициал «2ГИС советует обновить браузер»**, а не карточка фирмы (контент за JS).
Т.е. ratio 0.98 ввёл бы в заблуждение так же, как раньше word_count. 2gis httpx с текущим
UA НЕ даёт данных — остаётся браузер+resource-block (−94% байт) либо newer-UA/API-перехват.
Это второй пример, почему «зелёная метрика ≠ полезный контент».

## TL;DR
- **Топ-агрегаторы и реестры реально SSR-дружелюбны через httpx** — экономия
  **15–40×** байт И **столько же или больше** контента, чем у браузера. Это главный рычаг.
- **План был ОПТИМИСТИЧЕН по VK и 2GIS** — это SPA/walled, чистого SSR-текста они
  больше не отдают. Нужны адаптеры, а не наивный httpx-роутинг.
- **Instagram — login-wall во всех методах** (0 слов). План прав: не рендерить, брать
  @-хэндл из сниппета.
- **Resource-blocking местами даёт обратный эффект** (VK/IG: 0.1→3 МБ) — на анти-бот
  SPA `abort()` триггерит challenge. Блокировать только image/media/font и НЕ на соцсетях.
- Взвешенная по визитам модель: **−93%** на топ-доменах (267 МБ → 19 МБ / 427 визитов).

## Ключевые расхождения с PLAN.md
1. **VK ≠ «лёгкий SSR через m.vk.com».** m.vk.com отдаёт 385 КБ JS-страницы без `og:`-мет
   и без профильного текста (22 слова). httpx дёшев по байтам (58 КБ против 134 КБ–3 МБ
   браузера), но **извлечение требует адаптера** (inline-JSON / мобильное API), иначе
   данные не достаём. Самый ценный домен (147 визитов) — приоритет на адаптер.
2. **2gis ≠ «inline JSON SSR как Яндекс».** Это SPA: `/firm/<id>` через httpx = 3 КБ
   пустой shell. Контент приходит XHR'ом в `catalog.api.2gis.com`. Варианты: остаться
   на браузере+block (−94% байт, контент ок) ИЛИ перехват API (как в `yandex_maps.py`).
3. **Resource-blocking не универсален.** На обычных сайтах/агрегаторах режет 40–95%, но
   на VK/IG `route.abort()` триггерит challenge → байты РАСТУТ. Применять только к
   image/media/font и только на не-соцсетевом пути.
4. **«blocked»-маркеры (captcha/проверка) часто ложные** — hh/zoon/rusprofile отдали
   тысячи слов реального контента при наличии слова «проверка»/captcha-виджета в HTML.
   Критерий успеха — наличие контента, а не подстроки.

## Что подтвердилось из плана
- Resource-blocking как дешёвый универсальный рычаг на браузерном пути — да (с оговоркой #3).
- httpx-SSR fast-path для t.me (`/s/`), hh, zoon, prodoctorov, rusprofile, реестров — да, сильно.
- Per-domain cap + URL-dedup против «нор» (один сайт орг ×5–9) — обоснованно (см. spina/lianamed/ozerki ×5–9 в analyze_traffic).
- Instagram деприоритизировать — да.

## Рекомендация (приоритеты по ROI)
0. **СНАЧАЛА исправить brotli** (блокер для всего httpx-пути): в httpx-fetch слать
   `Accept-Encoding: gzip, deflate` (без `br`) ЛИБО `uv add brotli`. Иначе zoon/
   rubrikator/spravker/2gis/… молча отдают мусор. Это же поправить в
   `yandex_maps._HTTP_HEADERS`. **Без этого пункта остальное не работает.**
1. **httpx-SSR allowlist** — подтверждено чистым контентом после brotli-фикса:
   `t.me`(→/s/), `hh.ru`/`spb.hh.ru`, `zoon.ru`, `prodoctorov.ru`, `rusprofile.ru`,
   `rubrikator.org`, `orgzz.ru`, `spb.spravker.ru` + прочие реестры (checko/list-org).
   Роутинг в `tools.py:scrape_url`, парсер — reuse `html_to_text`. **Главный выигрыш.**
2. **t.me URL-rewrite** `t.me/X → t.me/s/X` (даёт БОЛЬШЕ контента, чем браузер: 324 vs 46 слов).
3. **Resource-block** (только image/media/font, НЕ на соцсетях) на оставшемся
   браузерном пути: **2gis** (httpx даёт лишь интерстициал → остаётся браузер, −94% байт)
   и сайты орг. Альтернатива для 2gis позже: newer-UA, чтобы обойти «обновите браузер»,
   или XHR-перехват `catalog.api.2gis.com` (как yandex).
4. **VK-адаптер** — отдельно (`site_adapters.py`): m.vk.com отдаёт стаб «браузер устарел»,
   генерик-текст бесполезен. Нужен парс inline-JSON / VK API. Высокая ценность (147
   визитов), средняя сложность. До адаптера VK лучше оставить на браузере, чем сломать.
5. **Instagram-стаб** — не рендерить, вернуть @-хэндл из URL/сниппета (login-wall везде).
6. **Per-domain visit cap (≈2–3) + URL-dedup** в `AgentState` — режет «норы».
7. Prompt-правки (авторитетные источники, анти-нора) — вторично.
