# A/B-валидация внедрения: OLD путь vs NEW (15 реальных орг)

**Дата:** 2026-06-09. Метод (см. `ab_replay.py`): для 15 орг из `data_750m/research_final`
взяты их РЕАЛЬНЫЕ посещённые URL и прогнаны через оба пути с замером трафика по проводу
и сохранности контента.

- **OLD** = поведение до изменений: браузер на КАЖДЫЙ URL, без resource-block, без httpx, без IG-стаба.
- **NEW** = то, что внедрено: `http_fetch` allowlist→httpx, IG→стаб, остальное→браузер+resource-block.

t.me пропущен (Telegram недоступен с этой машины без VPN — симметричный шум в обоих плечах).
Метрика контента — НАЛИЧИЕ по URL (регрессия = OLD дал ≥40 слов, NEW — нет), т.к. сырой
счётчик слов — артефакт (OLD-браузер отдаёт полный текст, NEW-httpx обрезан до 30k симв.,
а агент всё равно потребляет лишь 3.5k-символьную goal-выжимку).

## Как читать разброс 0.8×–9.8×
- **Высокий фактор (5–10×)** — у орг, чей трафик идёт на allowlist-домены (prodoctorov, hh,
  zoon, реестры): там браузер заменён на httpx (30–50× на странице).
- **Около 1× (0.8–1.3×)** — у орг, где доминирует **vk** (он by design остаётся на браузере,
  ~3 МБ/страница без изменений) + сайт орг (только resource-block). Случаи 0.8–0.9× —
  это в т.ч. разброс между двумя отдельными браузерными загрузками (OLD и NEW грузятся
  разными прокси/таймингом) и редкие reload'ы при resource-block.

## Что это значит
1. **Валидация пройдена:** −51% трафика, 0 регрессий контента на 15 орг. Качество не просело.
2. **vk — главный остаточный потребитель.** Реалистичная экономия на орг (2×) ограничена
   тем, что vk (147 визитов в прод-прогоне) оставлен на браузере. **VK-адаптер — следующий
   по ROI рычаг** (поднимет фактор существенно).
3. **Недооценка:** A/B изолирует смену метода фетча на одних и тех же URL. Внедрённые
   **per-domain cap + URL-dedup** дополнительно режут трафик, отсекая лишние визиты ЦЕЛИКОМ —
   здесь не учтено (оба плеча реплеят идентичный список URL). В реальном прогоне экономия выше.
