# Оптимизация скрейпа: снизить трафик/стоимость web_scrape

**Дата:** 2026-06-09. Контекст: полный прогон 132 орг съел **~2 ГБ трафика**. Цель —
срезать его в разы **без потери контента**, адресно по сайтам, разделив правки на
**harness-уровень** (детерминированные, агент не в курсе) и **prompt-уровень**
(поведение/выбор источника агентом).

---

## 1. Где «утекает» трафик
`web_scrape(url)` → `tools.py:scrape_url` → `SiteEnrichAction.execute` →
**Playwright `page.goto`** (`src/actions/site_enricher.py`). То есть **каждый** скрейп —
это полная загрузка страницы браузером: HTML + CSS + JS + картинки + шрифты + трекеры.
Это в **~10–50× тяжелее**, чем `httpx` GET одного HTML.

Для большинства нужных нам данных (текст карточки, контакты, реестр, вакансии) **JS не
нужен** — сайты отдают контент в SSR-HTML. Браузер реально нужен только для тяжёлых SPA.

## 3. Три рычага
1. **Resource-blocking в браузере** (где браузер всё же нужен) — рубить картинки/шрифты/
   медиа/css. Самый дешёвый и универсальный: −60–80% байт без потери текста, без изменения
   поведения агента.
2. **httpx-SSR fast-path** для allowlist-доменов (t.me, vk, 2gis, zoon, prodoctorov, hh,
   rusprofile, …) — заменить браузер на `httpx + selectolax/bs4`.
3. **Per-domain cap + dedup** — не бить один домен 5–9× на орг; не качать один URL дважды.

## 5. HARNESS vs PROMPT — что куда (главное)
Принцип: **всё механическое (КАК качать) — в harness** (детерминированно, не жрёт токены
агента, агент не может «забыть»). В prompt — **только выбор источника/суждение** (КУДА идти).

### → HARNESS-уровень (агент не меняет поведение, дешевеет исполнение)
1. **Resource-blocking в Playwright** (`site_enricher.py` / browser pool): перехват
   `route` — `abort()` для `image|media|font|stylesheet` и трекеров. Применять ко ВСЕМ
   браузерным скрейпам. Самый большой байт-выигрыш, нулевой риск. **Приоритет #1.**
2. **Domain→fetch роутинг** (`tools.py:scrape_url`): если домен в `HTTPX_SSR_ALLOWLIST` —
   качать `httpx + selectolax`, иначе браузер. Агент зовёт `web_scrape(url)` как прежде.
3. **URL-rewrite перед фетчем** (harness): `t.me/X`→`t.me/s/X`; `vk.com/X`→`m.vk.com/X`;
   `instagram.com/*` → не рендерить, вернуть лёгкий стаб с хэндлом.
4. **Per-domain visit cap на орг** (`agent.py`, рядом с `domain_fail_count`): >N визитов
   на домен → короткий отказ с подсказкой «этот домен исчерпан». N≈2–3.
5. **URL-dedup/кэш в рамках прогона** (`agent.py`/`state`): не качать идентичный URL дважды.
6. **Бюджет символов по классу домена** (`config.py`): реестрам больше, соцсетям меньше.

### → PROMPT-уровень (меняет ВЫБОР агента, требует суждения)
1. **Авторитетные источники по факту** (`system`/query): «ИНН → rusprofile/checko;
   вакансии → hh.ru; мед-рейтинги/врачи → prodoctorov; контакты/DM → vk/t.me». Сужает
   разброс заходов, поднимает польза/визит.
2. **Не открывать соцстраницу ради одного хэндла**: «@-хэндл бери из Я.карточки/сниппета
   выдачи; открывай соцсеть скрейпом только если нужен текст профиля/контакты внутри».
3. **Анти-нора по сайту орг**: «1–2 страницы с сайта орг достаточно; не обходи весь сайт»
   (дублирует harness-cap как мягкая подсказка).
4. **Агрегатор-first для структурных фактов**: «сначала 2gis/zoon/prodoctorov (структурно),
   потом сайт орг».

**Почему так:** #5/#1 (resource-block, роутинг, cap) — чисто механика, должны быть в
harness: дёшево, надёжно, не тратят ходы/контекст агента и работают даже если модель
«забудет» инструкцию. Prompt-правки нужны лишь там, где это РЕШЕНИЕ агента (какой источник
авторитетнее, стоит ли вообще лезть на IG) — это нельзя зашить детерминированно.

## 6. Реализация (где править)
- `src/actions/site_enricher.py` — resource-block (route interception) в браузерном пути.
- `src/actions/research/tools.py:scrape_url` — domain-роутинг httpx-SSR vs браузер;
  новый `_httpx_ssr_fetch(url)` (переиспользовать SSR-хелперы из `yandex_maps.py`:
  `_iter_big_blobs`, прокси-ротацию).
- Новый модуль `src/actions/research/site_adapters.py` — per-domain парсеры
  (t.me / vk / 2gis / zoon / prodoctorov / hh / rusprofile): URL-rewrite + извлечение.
- `src/actions/research/agent.py` — per-domain visit cap + URL-dedup в `AgentState`.
- `src/core/config.py` — `HTTPX_SSR_ALLOWLIST`, `SCRAPE_DOMAIN_VISIT_CAP`, бюджеты.
- `src/actions/research/research_agent_prompts.yaml` (`system`) — prompt-правки из §5.

## 7. Приоритеты и ожидаемая экономия
1. **Resource-block в браузере** — −60–80% байт на всех браузерных скрейпах, ~1 день, риск 0.
2. **httpx-SSR для t.me + vk + 2gis + zoon** (≈272 визита, 28% трафика) — заменяет браузер
   полностью на этих доменах. Высокий ROI.
3. **rusprofile/hh/prodoctorov httpx** — меньше объём, но высокая польза/визит.
4. **Per-domain cap + dedup** — убирает «норы» (один сайт ×5–9), −визиты ~10–15%.
5. **Prompt-правки** — польза/визит вверх, но вторично относительно harness-байтов.

Ориентир: harness-механика (#1–#2) реально способна срезать **бóльшую часть 2 ГБ**, т.к.
бьёт по самому тяжёлому — браузерным загрузкам на топ-доменах.

## 8. Риски / валидация
- httpx-SSR может ловить капчу/блок на части доменов → fallback на браузер (как в Я.картах).
- Соцсети меняют разметку → адаптеры держать изолированно (`site_adapters.py`), с фолбэком.
- **Валидация:** прогнать 15–20 орг старым путём vs новым; сравнить (а) трафик (МБ/орг),
  (б) заполняемость/польза карты — НЕ должна просесть, (в) время. Бенчмарк трафика снимать
  на уровне прокси/`SiteEnrichAction` (счётчик байт ответа).
