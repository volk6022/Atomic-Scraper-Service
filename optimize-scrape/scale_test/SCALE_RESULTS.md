# Scale test: стабильность ответов + трафик в масштабе

**Дата:** 2026-06-09. По каждому сайту — **100 запросов через резидентские прокси**,
циклом по **20 разным URL** (разные страницы = разные объёмы трафика). Замер: доля
успешных запросов, доля ответов с реальным контентом (site-specific валидатор), трафик
по проводу (`num_bytes_downloaded`), попытки/латентность. Concurrency=4.
Скрипты: `site_*.py` + общий `_common.py`; запуск всех — `run_all.py`. Сырьё: `results/*.json`.

> httpx шлёт `Accept-Encoding: gzip, deflate` (без brotli — иначе мусор, см. ../FINDINGS.md).

## Выводы
1. **Стабильность httpx-доменов высокая: 90–100% контента** на 100 запросах каждый,
   среднее число попыток 1.2–2.2 (в основном успех с первого прокси). Просадки 5–10% —
   транзиентные прокси-сбои, гасятся ротацией (6 ретраев). Метод стабилен в масштабе.
2. **Экономия трафика против браузера — подтверждена в масштабе:**
   на 700 запросах (7 доменов) браузер бы скачал **~946 МБ**, httpx — **38 МБ** =
   **25× / −96%**. По доменам: prodoctorov 50×, rubrikator 37×, hh 25×, rusprofile 20×,
   zoon 19×, spravker 13×, orgzz 11×.
3. **VK и 2gis: 100% запросов, 0% контента** — на 100 запросах КАЖДЫЙ раз стаб/интерстициал.
   Это окончательно подтверждает: их нельзя роутить в httpx (нужен браузер/адаптер).
   httpx на них только жжёт трафик впустую (vk 5.7 МБ/100 за нулевой контент).
4. **t.me — среда, а не метод:** Telegram недоступен с этой машины, когда трафик идёт
   не через VPN (даже через прокси). Метод httpx `/s/` валиден (доказан в FINDINGS),
   но `site_tme.py` нужно прогонять с VPN-машины. В батч (`run_all.py`) t.me не включён.
