# A/B — prefill Я.Карточки + отзывы (E1/E2/E3)

Парный A/B на 14 одиночных орг (фикснутый агент, mode=balanced).
Arm A = текущий query. Arm B = query + блок «ИЗВЕСТНЫЕ ДАННЫЕ из Я.Карты (как факт)»
+ отзывы (последние 6 мес). Данные prefill — `ab_test/context/{oid}.json`
(httpx, детерминированно). Сравнение: `ab_compare.py` → `ab_comparison.json`.

## Результат (sum по 14, парные B>A : B<A)
| поле | A → B | парно B>A:B<A | вердикт |
|---|---|---|---|
| **social** | 25 → **33** | 8:2 | **B лучше** |
| **phones** | 20 → **30** | 7:1 | **B лучше** |
| **problems_signals** | 28 → **37** | 7:4 | **B лучше** (из отзывов) |
| what_they_do | 14 → 14 | 0:0 | = |
| vacancies | 13 → 13 | 5:4 | = |
| scale_indicators | 34 → 30 | 6:6 | B чуть хуже |
| websites | 25 → 22 | 5:5 | B чуть хуже |
| sources | 88 → 74 | 4:8 | B меньше цитат |
| emails | 4 → 2 | 1:3 | шум (email почти нет) |

Стоимость: A turns 11.4 / serp 12.1 / scrape 7.0 / 294 с; B turns **9.9** / serp 13.6
/ scrape 7.1 / 325 с. Пустых карт: 0/0. B включил ≥ prefill-соцсетей: 10/12.

## Выводы
1. **B выигрывает на критичных для аутрича полях** — соцсети, телефоны,
   problems_signals (сигналы из отзывов). Это ровно то, что нужно для соц-DM-аутрича
   и персонализации предложения.
2. **Цикл короче** (turns 11.4→9.9): агент раньше получает полную картину.
3. Лёгкая просадка вторичных полей (sources/scale/websites) — агент «удовлетворяется»
   полной картой и меньше доисследует. Митигируется промптом («контакты закрыты —
   теперь тщательно добери scale/vacancies/email»).
4. **email почти отсутствует** (4 на 14 орг) — подтверждает: канал = соцсети, не email.

## Рекомендации для прода
1. **Принять prefill+отзывы (Arm B).** Чистый плюс по нужным полям + короче цикл.
2. **Соцсети/телефоны/часы — тянуть ДЕТЕРМИНИРОВАННО из Я.Карточки** (httpx, ~100 КБ),
   не полагаясь на LLM. LLM-ресёрч — только на gap-поля (email, сайты, вакансии,
   масштаб, what_they_do, tech для tech-орг).
3. Промпт-твик: prefill не должен снижать усилия по scale/sources — явно просить
   добрать их после контактов.

## Связь с E4 (инкрементальный submit)
A/B показал: prefill сокращает цикл и даёт **0 пустых карт** уже сейчас. Значит
ПОЛНЫЙ checkpoint-протокол НЕ нужен — достаточно упрощённого варианта:
**предзаполненная стартовая карта + подсказка по оставшимся пробелам + no-progress-
стоп**. Это и закрывает «не уходить в дебри / не упираться в лимит 25» без сложного
многократного submit.
